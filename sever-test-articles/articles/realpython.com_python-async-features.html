<p>Have you heard of asynchronous programming in Python? Are you curious to know more about Python async features and how you can use them in your work? Perhaps you&rsquo;ve even tried to write threaded programs and run into some issues. If you&rsquo;re looking to understand how to use Python async features, then you&rsquo;ve come to the right place.</p>
<p><strong>In this article, you&rsquo;ll learn:</strong></p>
<ul>
<li>What a <strong>synchronous program</strong> is</li>
<li>What an <strong>asynchronous program</strong> is</li>
<li>Why you might want to write an asynchronous program</li>
<li>How to use Python async features</li>
</ul>
<p>All of the example code in this article have been tested with Python 3.7.2. You can grab a copy to follow along by clicking the link below:</p>
<div class="alert alert-warning" role="alert"><p><strong>Download Code:</strong> <a href="https://realpython.com/optins/view/async-features/" class="alert-link" data-toggle="modal" data-target="#modal-async-features" data-focus="false">Click here to download the code you'll use</a> to learn about async features in Python in this tutorial.</p></div>
<div class="w-100 text-center js-needs-scaling" style="transform-origin: 0 0;"><div id="waldo-tag-4996"></div> <a class="small text-muted js-disclosure" href="/account/join/" rel="nofollow" style="display: none;"> <i aria-hidden="true" class="fa fa-info-circle"> </i> Remove ads</a></div><h2 id="understanding-asynchronous-programming">Understanding Asynchronous Programming</h2>
<p>A <strong>synchronous program</strong> is executed one step at a time. Even with conditional branching, loops and function calls, you can still think about the code in terms of taking one execution step at a time. When each step is complete, the program moves on to the next one.</p>
<p>Here are two examples of programs that work this way:</p>
<ul>
<li>
<p><strong>Batch processing programs</strong> are often created as synchronous programs. You get some input, process it, and create some output. Steps follow one after the other until the program reaches the desired output. The program only needs to pay attention to the steps and their order.</p>
</li>
<li>
<p><strong>Command-line programs</strong> are small, quick processes that run in a terminal. These scripts are used to create something, transform one thing into something else, generate a report, or perhaps list out some data. This can be expressed as a series of program steps that are executed sequentially until the program is done.</p>
</li>
</ul>
<p>An <strong>asynchronous program</strong> behaves differently. It still takes one execution step at a time. The difference is that the system may not wait for an execution step to be completed before moving on to the next one.</p>
<p>This means that the program will move on to future execution steps even though a previous step hasn&rsquo;t yet finished and is still running elsewhere. This also means that the program knows what to do when a previous step does finish running.</p>
<p>Why would you want to write a program in this manner? The rest of this article will help you answer that question and give you the tools you need to elegantly solve interesting asynchronous problems.</p>
<h3 id="building-a-synchronous-web-server">Building a Synchronous Web Server</h3>
<p>A web server&rsquo;s basic unit of work is, more or less, the same as batch processing. The server will get some input, process it, and create the output. Written as a synchronous program, this would create a working web server.</p>
<p>It would also be an <em>absolutely terrible</em> web server.</p>
<p>Why? In this case, one unit of work (input, process, output) is not the only purpose. The real purpose is to handle hundreds or even thousands of units of work as quickly as possible. This can happen over long periods of time, and several work units may even arrive all at once.</p>
<p>Can a synchronous web server be made better? Sure, you could optimize the execution steps so that all the work coming in is handled as quickly as possible. Unfortunately, there are limitations to this approach. The result could be a web server that doesn&rsquo;t respond fast enough, can&rsquo;t handle enough work, or even one that times out when work gets stacked up.</p>
<div class="alert alert-primary" role="alert">
<p><strong>Note:</strong> There are other limitations you might see if you tried to optimize the above approach. These include network speed, file IO speed, database query speed, and the speed of other connected services, to name a few. What these all have in common is that they are all IO functions. All of these items are orders of magnitude slower than the CPU&rsquo;s processing speed.</p>
</div>
<p>In a synchronous program, if an execution step starts a database query, then the CPU is essentially idle until the database query is returned. For batch-oriented programs, this isn&rsquo;t a priority most of the time. Processing the results of that IO operation is the goal. Often, this can take longer than the IO operation itself. Any optimization efforts would be focused on the processing work, not the IO.</p>
<p>Asynchronous programming techniques allow your programs to take advantage of relatively slow IO processes by freeing the CPU to do other work.</p>
<h3 id="thinking-differently-about-programming">Thinking Differently About Programming</h3>
<p>When you start trying to understand asynchronous programming, you might see a lot of discussion about the importance of blocking, or writing <a href="https://medium.com/vaidikkapoor/understanding-non-blocking-i-o-with-python-part-1-ec31a2e2db9b">non-blocking code</a>. (Personally, I struggled to get a good grasp of these concepts from the people I asked and the documentation I read.)</p>
<p>What is non-blocking code? What&rsquo;s blocking code, for that matter? Would the answers to these questions help you write a better web server? If so, how could you do it? Let&rsquo;s find out!</p>
<p>Writing asynchronous programs requires that you think differently about programming. While this new way of thinking can be hard to wrap your head around, it&rsquo;s also an interesting exercise. That&rsquo;s because the real world is almost entirely asynchronous, and so is how you interact with it.</p>
<p>Imagine this: you&rsquo;re a parent trying to do several things at once. You have to balance the checkbook, do the laundry, and keep an eye on the kids. Somehow, you&rsquo;re able to do all of these things at the same time without even thinking about it! Let&rsquo;s break it down:</p>
<ul>
<li>
<p>Balancing the checkbook is a <strong>synchronous</strong> task. One step follows another until it&rsquo;s done. You&rsquo;re doing all the work yourself.</p>
</li>
<li>
<p>However, you can break away from the checkbook to do laundry. You unload the dryer, move clothes from the washer to the dryer, and start another load in the washer.</p>
</li>
<li>
<p>Working with the washer and dryer is a synchronous task, but the bulk of the work happens <em>after</em> the washer and dryer are started. Once you&rsquo;ve got them going, you can walk away and get back to the checkbook task. At this point, the washer and dryer tasks have become <strong>asynchronous</strong>. The washer and dryer will run independently until the buzzer goes off (notifying you that the task needs attention).</p>
</li>
<li>
<p>Watching your kids is another asynchronous task. Once they are set up and playing, they can do so independently for the most part. This changes when someone needs attention, like when someone gets hungry or hurt. When one of your kids yells in alarm, you react. The kids are a long-running task with high priority. Watching them supersedes any other tasks you might be doing, like the checkbook or laundry.</p>
</li>
</ul>
<p>These examples can help to illustrate the concepts of blocking and non-blocking code. Let&rsquo;s think about this in programming terms. In this example, you&rsquo;re like the CPU. While you&rsquo;re moving the laundry around, you (the CPU) are busy and blocked from doing other work, like balancing the checkbook. But that&rsquo;s okay because the task is relatively quick.</p>
<p>On the other hand, starting the washer and dryer does not block you from performing other tasks. It&rsquo;s an asynchronous function because you don&rsquo;t have to wait for it to finish. Once it&rsquo;s started, you can go back to something else. This is called a context switch: the context of what you&rsquo;re doing has changed, and the machine&rsquo;s buzzer will notify you sometime in the future when the laundry task is complete.</p>
<p>As a human, this is how you work all the time. You naturally juggle multiple things at once, often without thinking about it. As a developer, the trick is how to translate this kind of behavior into code that does the same kind of thing.</p>
<div class="w-100 text-center js-needs-scaling" style="transform-origin: 0 0;"><div id="waldo-tag-4998"></div> <a class="small text-muted js-disclosure" href="/account/join/" rel="nofollow" style="display: none;"> <i aria-hidden="true" class="fa fa-info-circle"> </i> Remove ads</a></div><h2 id="programming-parents-not-as-easy-as-it-looks">Programming Parents: Not as Easy as It Looks!</h2>
<p>If you recognize yourself (or your parents) in the example above, then that&rsquo;s great! You&rsquo;ve got a leg up in understanding asynchronous programming. Again, you&rsquo;re able to switch contexts between competing tasks fairly easily, picking up some tasks and resuming others. Now you&rsquo;re going to try and program this behavior into virtual parents!</p>
<h3 id="thought-experiment-1-the-synchronous-parent">Thought Experiment #1: The Synchronous Parent</h3>
<p>How would you create a parent program to do the above tasks in a completely synchronous manner? Since watching the kids is a high-priority task, perhaps your program would do just that. The parent watches over the kids while waiting for something to happen that might need their attention. However, nothing else (like the checkbook or laundry) would get done in this scenario.</p>
<p>Now, you can re-prioritize the tasks any way you want, but only one of them would happen at any given time. This is the result of a synchronous, step-by-step approach. Like the synchronous web server described above, this would work, but it might not be the best way to live. The parent wouldn&rsquo;t be able to complete any other tasks until the kids fell asleep. All other tasks would happen afterward, well into the night. (A couple of weeks of this and many real parents might jump out the window!)</p>
<h3 id="thought-experiment-2-the-polling-parent">Thought Experiment #2: The Polling Parent</h3>
<p>If you used <a href="https://en.wikipedia.org/wiki/Polling_(computer_science)">polling</a>, then you could change things up so that multiple tasks are completed. In this approach, the parent would periodically break away from the current task and check to see if any other tasks need attention.</p>
<p>Let&rsquo;s make the polling interval something like fifteen minutes. Now, every fifteen minutes your parent checks to see if the washer, dryer or kids need any attention. If not, then the parent can go back to work on the checkbook. However, if any of those tasks <em>do</em> need attention, then the parent will take care of it before going back to the checkbook. This cycle continues on until the next timeout out of the polling loop.</p>
<p>This approach works as well since multiple tasks are getting attention. However, there are a couple of problems:</p>
<ol>
<li>
<p><strong>The parent may spend a lot of time checking on things that don&rsquo;t need attention:</strong> The washer and dryer haven&rsquo;t yet finished, and the kids don&rsquo;t need any attention unless something unexpected happens.</p>
</li>
<li>
<p><strong>The parent may miss completed tasks that do need attention:</strong> For instance, if the washer finished its cycle at the beginning of the polling interval, then it wouldn&rsquo;t get any attention for up to fifteen minutes! What&rsquo;s more, watching the kids is supposedly the highest priority task. They couldn&rsquo;t tolerate fifteen minutes with no attention when something might be going drastically wrong.</p>
</li>
</ol>
<p>You could address these issues by shortening the polling interval, but now your parent (the CPU) would be spending more time context switching between tasks. This is when you start to hit a point of diminishing returns. (Once again, a couple of weeks living like this and, well&hellip; See the previous comment about windows and jumping.)</p>
<h3 id="thought-experiment-3-the-threading-parent">Thought Experiment #3: The Threading Parent</h3>
<p>&ldquo;If I could only clone myself&hellip;&rdquo; If you&rsquo;re a parent, then you&rsquo;ve probably had similar thoughts! Since you&rsquo;re programming virtual parents, you can essentially do this by using threading. This is a mechanism that allows multiple sections of one program to run at the same time. Each section of code that runs independently is known as a thread, and all threads share the same memory space.</p>
<p>If you think of each task as a part of one program, then you can separate them and run them as threads. In other words, you can &ldquo;clone&rdquo; the parent, creating one instance for each task: watching the kids, monitoring the washer, monitoring the dryer, and balancing the checkbook. All of these &ldquo;clones&rdquo; are running independently.</p>
<p>This sounds like a pretty nice solution, but there are some issues here as well. One is that you&rsquo;ll have to explicitly tell each parent instance what to do in your program. This can lead to some problems since all instances share everything in the program space.</p>
<p>For example, say that Parent A is monitoring the dryer. Parent A sees that the clothes are dry, so they take control of the dryer and begin unloading the clothes. At the same time, Parent B sees that the washer is done, so they take control of the washer and begin removing clothes. However, Parent B also needs to take control of the dryer so they can put the wet clothes inside. This can&rsquo;t happen, because Parent A currently has control of the dryer.</p>
<p>After a short while, Parent A has finished unloading clothes. Now they want to take control of the washer and start moving clothes into the empty dryer. This can&rsquo;t happen, either, because Parent B currently has control of the washer!</p>
<p>These two parents are now <a href="https://realpython.com/intro-to-python-threading/#deadlock">deadlocked</a>. Both have control of their own resource <em>and</em> want control of the other resource. They&rsquo;ll wait forever for the other parent instance to release control. As the programmer, you&rsquo;d have to write code to work this situation out.</p>
<div class="alert alert-primary" role="alert">
<p><strong>Note:</strong> Threaded programs allow you to create multiple, parallel paths of execution that all share the same memory space. This is both an advantage and a disadvantage. Any memory shared between threads is subject to one or more threads trying to use the same shared memory at the same time. This can lead to data corruption, data read in an invalid state, and data that&rsquo;s just messy in general.</p>
<p>In threaded programming, the context switch happens under system control, not the programmer. The system controls when to switch contexts and when to give threads access to shared data, thereby changing the context of how the memory is being used. All of these kinds of problems are manageable in threaded code, but it&rsquo;s difficult to get right, and hard to debug when it&rsquo;s wrong.</p>
</div>
<p>Here&rsquo;s another issue that might arise from threading. Suppose that a child gets hurt and needs to be taken to urgent care. Parent C has been assigned the task of watching over the kids, so they take the child right away. At the urgent care, Parent C needs to write a fairly large check to cover the cost of seeing the doctor.</p>
<p>Meanwhile, Parent D is at home working on the checkbook. They&rsquo;re unaware of this large check being written, so they&rsquo;re very surprised when the family checking account is suddenly overdrawn!</p>
<p>Remember, these two parent instances are working within the same program. Otherwise, you&rsquo;d need to provide some kind of locking mechanism so that the checkbook resource can only be used by one parent at a time, with updates.</p>

<h3 id="simple-cooperative-concurrency">Simple Cooperative Concurrency</h3>
<p>The next version of the program allows the two tasks to work together. Adding a <code>yield</code> statement means the loop will yield control at the specified point while still maintaining its context. This way, the yielding task can be restarted later.</p>
<p>The <code>yield</code> statement turns <code>task()</code> into a <a href="https://realpython.com/introduction-to-python-generators/">generator</a>. A generator function is called just like any other function in Python, but when the <code>yield</code> statement is executed, control is returned to the caller of the function. This is essentially a context switch, as control moves from the generator function to the caller.</p>
<p>The interesting part is that control can be given <em>back</em> to the generator function by calling <code>next()</code> on the generator. This is a context switch back to the generator function, which picks up execution with all function variables that were defined before the <code>yield</code> still intact.</p>
<p>The <code>while</code> loop in <code>main()</code> takes advantage of this when it calls <code>next(t)</code>. This statement restarts the task at the point where it previously yielded. All of this means that you&rsquo;re in control when the context switch happens: when the <code>yield</code> statement is executed in <code>task()</code>.</p>
<p>This is a form of cooperative multitasking. The program is yielding control of its current context so that something else can run. In this case, it allows the <code>while</code> loop in <code>main()</code> to run two instances of <code>task()</code> as a generator function. Each instance consumes work from the same queue. This is sort of clever, but it&rsquo;s also a lot of work to get the same results as the first program. <h3 id="cooperative-concurrency-with-non-blocking-calls">Cooperative Concurrency With Non-Blocking Calls</h3>
<p>The next version of the program has been modified quite a bit. It makes use of Python async features using <a href="https://realpython.com/async-io-python/">asyncio/await</a> provided in Python 3.</p>
<p>The <code>time</code> and <code>queue</code> modules have been replaced with the <code>asyncio</code> package. This gives your program access to asynchronous friendly (non-blocking) sleep and queue functionality. The change to <code>task()</code> defines it as asynchronous with the addition of the <code>async</code> prefix on line 4. This indicates to Python that the function will be asynchronous.</p>
<p>The other big change is removing the <code>time.sleep(delay)</code> and <code>yield</code> statements, and replacing them with <code>await asyncio.sleep(delay)</code>. This creates a non-blocking delay that will perform a context switch back to the caller <code>main()</code>.</p>
<p>The <code>while</code> loop inside <code>main()</code> no longer exists. Instead of <code>task_array</code>, there&rsquo;s a call to <code>await asyncio.gather(...)</code>. This tells <code>asyncio</code> two things:</p>
<ol>
<li>Create two tasks based on <code>task()</code> and start running them.</li>
<li>Wait for both of these to be completed before moving forward.</li>
</ol>
<p>The last line of the program <code>asyncio.run(main())</code> runs <code>main()</code>. This creates what&rsquo;s known as an <a href="https://realpython.com/lessons/asyncio-event-loop/">event loop</a>). It&rsquo;s this loop that will run <code>main()</code>, which in turn will run the two instances of <code>task()</code>.</p>
<p>The event loop is at the heart of the Python async system. It runs all the code, including <code>main()</code>. When task code is executing, the CPU is busy doing work. When the <code>await</code> keyword is reached, a context switch occurs, and control passes back to the event loop. The event loop looks at all the tasks waiting for an event (in this case, an <code>asyncio.sleep(delay)</code> timeout) and passes control to a task with an event that&rsquo;s ready.</p>
<p><code>await asyncio.sleep(delay)</code> is non-blocking in regards to the CPU. Instead of waiting for the delay to timeout, the CPU registers a sleep event on the event loop task queue and performs a context switch by passing control to the event loop. The event loop continuously looks for completed events and passes control back to the task waiting for that event. In this way, the CPU can stay busy if work is available, while the event loop monitors the events that will happen in the future.</p>
<div class="alert alert-primary" role="alert">
<p><strong>Note:</strong> An asynchronous program runs in a single thread of execution. The context switch from one section of code to another that would affect data is completely in your control. This means you can atomize and complete all shared memory data access before making a context switch. This simplifies the shared memory problem inherent in threaded code.</p>
</div>

<p>Because the CPU is so fast, this example could likely create as many tasks as there are URLs. In this case, the program&rsquo;s run time would be that of the single slowest URL retrieval.</p>

<p>This article has given you the tools you need to start making asynchronous programming techniques a part of your repertoire. Using Python async features gives you programmatic control of when context switches take place. This means that many of the tougher issues you might see in threaded programming are easier to deal with.</p>
<p>Asynchronous programming is a powerful tool, but it isn&rsquo;t useful for every kind of program. If you&rsquo;re writing a program that calculates pi to the millionth decimal place, for instance, then asynchronous code won&rsquo;t help you. That kind of program is CPU bound, without much IO. However, if you&rsquo;re trying to implement a server or a program that performs IO (like file or network access), then using Python async features could make a huge difference.</p>

(function(document, history, location) {

var anchorScrolls = {
ANCHOR_REGEX: /^#[^ ]+$/,
OFFSET_HEIGHT_PX: 120,

/**
* Establish events, and fix initial scroll position if a hash is provided.
*/
init: function() {
this.scrollToCurrent();
window.addEventListener('hashchange', this.scrollToCurrent.bind(this));
document.body.addEventListener('click', this.delegateAnchors.bind(this));
},

/**
* Return the offset amount to deduct from the normal scroll position.
* Modify as appropriate to allow for dynamic calculations
*/
getFixedOffset: function() {
return this.OFFSET_HEIGHT_PX;
},

/**
* If the provided href is an anchor which resolves to an element on the
* page, scroll to it.
* @param {String} href
* @return {Boolean} - Was the href an anchor.
*/
scrollIfAnchor: function(href, pushToHistory) {
var match, rect, anchorOffset;

if(!this.ANCHOR_REGEX.test(href)) {
return false;
}

match = document.getElementById(href.slice(1));

if(match) {
rect = match.getBoundingClientRect();
anchorOffset = window.pageYOffset + rect.top - this.getFixedOffset();
window.scrollTo(window.pageXOffset, anchorOffset);

// Add the state to history as-per normal anchor links
if(HISTORY_SUPPORT && pushToHistory) {
history.pushState({}, document.title, location.pathname + href);
}
}

return !!match;
},

/**
* Attempt to scroll to the current location's hash.
*/
scrollToCurrent: function() {
this.scrollIfAnchor(window.location.hash);
},

/**
* If the click event's target was an anchor, fix the scroll position.
*/
delegateAnchors: function(e) {
var elem = e.target;

if(
elem.nodeName === 'A' &&
this.scrollIfAnchor(elem.getAttribute('href'), true)
) {
e.preventDefault();
}
}
};

window.addEventListener(
'DOMContentLoaded', anchorScrolls.init.bind(anchorScrolls)
);
})(window.document, window.history, window.location);
